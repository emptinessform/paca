# 배포 문서

Paca는 [`deploy/`](../../deploy/README.md) 아래에 세 가지 Docker Compose 진입점을 제공합니다:

- 로컬 개발을 위한 `docker-compose.dev.yml`;
- 프로덕션 지향 단일 호스트 배포를 위한 `docker-compose.prod.yml`;
- 종단 간 테스트 자동화를 위한 `docker-compose.e2e.yml`.

## 분리되어 있는 이유

개발과 프로덕션은 서로 다른 목표를 가집니다:

- 개발은 빠른 온보딩, 검사 용이성, 로컬 피드백에 최적화되어 있습니다;
- 프로덕션은 명시적 구성, 이미지 기반 롤아웃, 운영자 제어에 최적화되어 있습니다.

이들을 분리해 두는 것이 더 깔끔한 오픈소스 기본값입니다. 이는 로컬 가정을 프로덕션 경로에 하드코딩하는 것을 방지하고, 기여자가 저장소를 이해하기 쉽게 만듭니다.

## 개발용 Compose

개발용 compose 파일은 다음을 프로비저닝합니다:

- PostgreSQL;
- Valkey;
- MinIO(파일 첨부를 위한 S3 호환 객체 저장소);
- 필요에 따라 인프라 서비스와 함께 실행할 수 있는 선택적 `api` 및 `web` 서비스 컨테이너.

이는 두 가지 워크플로를 지원합니다:

- Docker에서 인프라만 실행하고 애플리케이션 서비스는 호스트에서 시작하기;
- 빠른 종단 간 테스트를 위해 전체 스택을 Docker에서 실행하기.

## 프로덕션용 Compose

프로덕션용 compose 파일은 의도적으로 셀프 호스팅이 가능하도록 설계되었습니다:

- 웹 및 API 컨테이너를 정의합니다;
- 완전한 단일 호스트 스택을 위해 PostgreSQL과 Valkey를 포함합니다;
- 환경 변수와 명명된 볼륨을 통해 구성을 명시적으로 유지합니다;
- 기본적으로 웹 및 API 서비스를 게시합니다.

이는 더 나은 오픈소스 기준선을 만듭니다: 사용자는 전체 플랫폼을 즉시 실행할 수 있으며, 관리형 인프라를 갖춘 운영자는 연결 설정을 변경하여 번들된 서비스를 외부에서 호스팅되는 등가물로 교체할 수 있습니다.

## 객체 저장소

모든 환경은 S3 호환 객체 저장소인 MinIO와 함께 제공되므로, AWS 계정 없이도 파일 첨부가 기본적으로 동작합니다. API 서비스는 저장소 제공자에 구애받지 않습니다: AWS S3로 전환하려면 몇 개의 환경 변수만 변경하면 됩니다.

프로덕션에서는 MinIO가 기본적으로 실행됩니다. AWS S3를 사용할 때 MinIO 컨테이너를 억제하려면 `docker compose up` 명령에 `--scale minio=0`을 전달하세요.

| 시나리오 | 추가 플래그 | MinIO 컨테이너 |
|---|---|---|
| 셀프 호스팅(기본값) | _(없음)_ | 시작됨 |
| AWS S3 | `--scale minio=0` | 시작되지 않음 |

| 변수 | 기본값 | 설명 |
|---|---|---|
| `STORAGE_PROVIDER` | `minio` | `minio`(번들) 또는 `s3`(AWS S3) |
| `STORAGE_ENDPOINT` | `minio:9000` | 커스텀 엔드포인트; 기본 AWS 리전 엔드포인트를 사용하려면 비워 두세요 |
| `STORAGE_REGION` | `us-east-1` | S3 리전 |
| `STORAGE_BUCKET` | `paca` | 버킷 이름 |
| `STORAGE_ACCESS_KEY_ID` | — | 액세스 키 / MinIO 루트 사용자 |
| `STORAGE_SECRET_ACCESS_KEY` | — | 시크릿 키 / MinIO 루트 비밀번호 |
| `STORAGE_USE_SSL` | `false` | HTTPS로 연결할 때 `true`로 설정하세요 |

업로드와 다운로드 모두에 사전 서명된(presigned) URL이 사용되므로, 객체 저장소가 외부에 공개되는 일은 없습니다. 클라이언트는 단기 URL(업로드는 1시간, 다운로드는 15분)을 받아 저장소 백엔드와 직접 통신하며, API 서비스는 데이터 플레인 바깥에 유지됩니다.
